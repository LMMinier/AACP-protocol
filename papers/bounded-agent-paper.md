# Bounded-Resource Local-First LLM Agents: Serialized Specialist Routing and Deterministic Authority Guardrails

**Luis M. Minier**  
Independent Researcher, New York, NY  
**Version:** 1.0 technical report, June 20, 2026

## Abstract

Local-first language-model agents promise privacy, offline operation, and user control, but commodity hardware creates a difficult systems trade-off. A single small model is inexpensive but uneven across code, reasoning, and conversational tasks; loading several specialist models concurrently can exceed the available memory; and prompt-only safety mechanisms remain vulnerable when untrusted data and trusted instructions share one context. This paper presents a bounded-resource agent architecture that combines **serialized specialist routing** with **deterministic authority guardrails**. A router selects a task profile, a residency manager enforces an unload barrier before the selected specialist is loaded, route-specific inference parameters are applied, and all persistent or external effects pass through an authority-aware runtime rather than being decided by model text alone.

The architecture provides two conditional guarantees. First, when the backend enforces complete serialized residency, peak model memory is bounded by the largest specialist rather than the sum of all specialists. Second, when authority metadata is trusted and every sensitive sink is completely mediated, an untrusted context segment cannot authorize an effect outside its policy set, regardless of the wording or encoding of that segment. The accompanying artifact includes an Ollama-compatible measurement harness that records loaded-model sets, host memory over time, latency, throughput, route correctness, and validity flags. Five harness unit tests and a deterministic self-test pass in the development environment. Existing AACP artifacts report complete passes on their versioned internal authority, poisoning, provenance, and detector suites; these results support the guardrail mechanism but do not establish broad real-world security.

This report deliberately makes no unmeasured throughput, capability, or benchmark-superiority claim. Target-hardware results must be generated by the included harness and committed with raw JSON/CSV/SVG artifacts before such claims are added. The contribution is therefore a precise architecture, its conditional invariants, and a reproducible evaluation path—not a claim that orchestration turns a weak model into a universally strong one.

## 1. Introduction

Small and quantized language models can run on consumer laptops, desktops, and edge devices. This improves privacy and availability because prompts and model outputs need not leave the machine. The same constraint that makes local inference attractive—limited memory and compute—also limits model capability. A sub-billion-parameter model may respond quickly but perform poorly on code or multi-step reasoning. A larger coding specialist may be useful for programming but unnecessarily slow for short transformations. Keeping every specialist resident avoids reload latency but can exhaust memory.

Model routing addresses part of this problem. Prior work routes requests between strong and weak models to optimize cost or quality, and inference systems reduce the memory burden of large individual models. Those approaches do not, by themselves, provide a local peak-memory invariant across multiple full specialists, nor do they prevent an untrusted retrieved document from attempting to drive a tool, write long-term memory, or modify policy.

Prompt injection is especially important for agents. An ordinary chatbot can emit a bad answer; an agent can send email, write files, call APIs, or persist corrupted instructions. Greshake et al. showed that retrieved content can remotely manipulate LLM-integrated applications because instructions and data are blurred in a shared context. AgentDojo and ToolEmu further demonstrate that tool-using agents remain difficult to secure and evaluate. CaMeL responds by separating control and data flow and enforcing capabilities outside the model. The architecture in this paper adopts the same high-level systems lesson: **model output is a proposal, not authority**.

We study the following question:

> Can a local agent expose several capabilities under a fixed memory budget while preventing low-authority context from directly causing sensitive effects?

The proposed answer has four parts:

1. **Deterministic or learned task routing** selects a specialist profile.
2. **Serialized residency** permits at most one specialist model allocation at a time.
3. **Route-aware inference contracts** bind decoding parameters to task classes instead of using one global profile.
4. **Deterministic authority guardrails** mediate tools, persistence, memory, and policy changes outside the model.

The contributions are:

- a formal single-residency invariant and peak-memory upper bound;
- a complete-mediation authority invariant for sensitive effects;
- a generalized architecture that does not depend on a particular product UI;
- an Ollama-compatible measurement harness with raw artifacts and fail-closed validity checks;
- an honest separation between proven architectural properties, artifact-tested behavior, and still-unmeasured hardware performance.

## 2. Scope, Assumptions, and Non-Claims

### 2.1 Scope

The design targets a single-user local agent running multiple quantized specialists through a local model server. The agent may offer chat, code, mathematical, retrieval, voice, image, or automation capabilities, but this paper focuses on text-model routing and effect authorization.

### 2.2 Assumptions

The guarantees in this paper require:

- the router and guardrail runtime execute outside model-generated text;
- authority labels are assigned by trusted application code, not accepted from untrusted content;
- every sensitive effect passes through the same enforcement boundary;
- the model backend accurately reports loaded models;
- an unload operation completes before the next load begins;
- operating-system and backend allocations not directly controlled by the application are included in a measured transient term.

### 2.3 Non-claims

This work does **not** claim:

- that a small model becomes equivalent to a frontier model;
- that routing improves every benchmark;
- that heuristic routing is optimal;
- that local execution is automatically secure;
- that pattern detection alone solves prompt injection;
- that the present internal attack corpora establish production security;
- that the architecture has a measured speed or memory advantage on hardware for which raw results are absent.

These limits are not cosmetic. They define the boundary between an architectural guarantee and an empirical claim.

## 3. Related Work

### 3.1 Model routing and cascades

FrugalGPT studies prompt adaptation, model approximation, and model cascades for reducing paid API cost while preserving quality. RouteLLM learns routers from preference data to choose between stronger and weaker models. Both establish that per-request model selection can improve a cost-quality trade-off. The present work differs in its primary constraint: local peak memory under serialized residency, coupled to an external authority boundary.

### 3.2 Resource-efficient local inference

PowerInfer exploits activation locality to distribute hot and cold neurons across GPU and CPU. PowerInfer-2 extends memory-aware scheduling to smartphones and models larger than device memory. These systems optimize execution of a large model. The present architecture instead coordinates several complete specialists and provides a bound when only one may be resident. The two approaches are complementary: a specialist may itself run on PowerInfer, llama.cpp, Ollama, or another optimized backend.

### 3.3 Agent safety and prompt injection

Indirect prompt injection exploits the model's inability to reliably distinguish data from instructions. AgentDojo provides a dynamic benchmark for prompt-injection attacks and defenses in tool-using agents. ToolEmu emulates tool environments to expose high-impact failures. CaMeL proposes a protective system layer that extracts control and data flow from the trusted query and uses capabilities to constrain unauthorized flows. The authority guardrail in this paper belongs to this systems-oriented family: it does not ask the model to decide whether it should obey an untrusted segment.

### 3.4 Positioning

| System | Per-request model routing | Local/offline compatible | Explicit peak-RAM invariant across specialists | Deterministic sink authorization | Auditable decision receipts |
|---|---:|---:|---:|---:|---:|
| FrugalGPT | Yes | Not its focus | No | No | No |
| RouteLLM | Yes | Possible | No | No | No |
| PowerInfer / PowerInfer-2 | No | Yes | Single-model optimization | No | No |
| AgentDojo | Evaluation only | Possible | No | Evaluates defenses | Benchmark logs |
| CaMeL | No | Possible | No | Yes, capability/data-flow based | Implementation dependent |
| **This architecture** | Yes | Yes | **Yes, conditional on serialized residency** | **Yes, authority/effect based** | **Yes** |

The novelty is not the isolated existence of routers, small models, or capability checks. It is the integration of a strict residency invariant, task-specific inference contracts, and complete-mediation authority policy into one bounded local runtime with a reproducible measurement artifact.

## 4. System Model

Let the set of task routes be

\[
\mathcal{R}=\{r_{fast}, r_{chat}, r_{code}, r_{math}, \ldots\}.
\]

Each route maps to a profile

\[
P(r)=(m_r, \theta_r, c_r, \rho_r),
\]

where \(m_r\) is the specialist model, \(\theta_r\) represents decoding parameters, \(c_r\) is the context limit, and \(\rho_r\) is an optional policy or risk profile.

An input request \(x\) is mapped by a router

\[
f: x \rightarrow r.
\]

The router may be heuristic, learned, or hybrid. The reference harness contains a deterministic baseline so results are reproducible without a routing model.

The runtime state is

\[
S=(r, m, C, A, Q),
\]

where \(r\) is the selected route, \(m\) is the resident model or null, \(C\) is typed context, \(A\) is the authority policy, and \(Q\) is the audit log.

A request is processed as:

1. classify the request into route \(r\);
2. acquire the model-residency lock;
3. unload every non-selected model;
4. wait until the backend reports no conflicting resident model;
5. invoke \(m_r\) with \(\theta_r\);
6. convert proposed actions into typed effect requests;
7. authorize or reject each effect under \(A\);
8. emit a receipt into \(Q\);
9. optionally release or retain \(m_r\) under a bounded keep-alive policy.

## 5. Serialized Specialist Routing

### 5.1 Why serialization matters

If all specialists are resident, model memory is approximately additive. For model footprints \(M_1,\ldots,M_n\), concurrent residency approaches

\[
M_{models}\approx \sum_{i=1}^{n}M_i.
\]

Under strict serialized residency, model memory approaches

\[
M_{models}\approx \max_i M_i + \delta,
\]

where \(\delta\) captures backend overlap, allocator fragmentation, page cache, or delayed release.

### 5.2 Residency protocol

The residency manager uses an unload barrier:

```text
route(request) -> selected model
acquire residency lock
for each loaded model != selected model:
    request unload
wait until backend reports no conflicting model
invoke selected model
sample memory and loaded-model state throughout generation
release lock
```

A timeout fails the request rather than silently allowing overlap. This is essential: an intended serialization policy is not evidence that serialization occurred.

### 5.3 Route-aware inference

One global decoding profile is an avoidable source of failures. Code generation often benefits from low temperature and conservative repetition settings. Open-ended chat can tolerate more diversity. Arithmetic and structured extraction should minimize stochasticity. The profile is therefore selected together with the model.

This is not a claim that fixed settings are universally best. It is a contract that makes the active settings explicit, testable, and attributable to a route.

## 6. Deterministic Authority Guardrails

### 6.1 Typed context

Every context segment carries trusted metadata including source, authority, provenance, and whether it is allowed to influence specific effects. A representative authority order is:

```text
protocol root
system
developer
authenticated user
user content / delegated data
tool output / retrieved external
generated intermediate
unknown untrusted
```

Authority is not inferred from phrases such as “SYSTEM MESSAGE” inside retrieved text. It is assigned by the application boundary that introduced the segment.

### 6.2 Effects and sinks

Sensitive effects include:

- tool invocation;
- shell or code execution;
- file, database, or memory writes;
- network requests and message sending;
- credential access;
- policy changes;
- persistence and trust delegation.

For authority class \(a\), define the allowed-effect set \(E(a)\). An attempted effect \(e\) is permitted only when

\[
e \in E(a)
\]

and any additional conditions, such as explicit user confirmation, are satisfied.

### 6.3 Detector as a secondary signal

Textual or semantic injection detectors may still provide useful risk evidence. They can trigger quarantine, summarization, or additional confirmation. They are not the root of authority. Even a detector false negative must not let retrieved text acquire tool authority.

### 6.4 Complete mediation

The guardrail is effective only if every sensitive sink is mediated. A protected email tool alongside an unprotected raw HTTP client is not complete mediation. The implementation must therefore centralize effect execution and prohibit direct model-to-sink paths.

## 7. Formal Properties

### Theorem 1: Peak-memory bound under strict single residency

Let:

- \(B\) be non-model base runtime memory;
- \(G\) be guardrail and audit memory;
- \(R\) be router memory;
- \(F_i\) be the full runtime footprint of specialist \(i\), including weights, KV cache, and workspace for the tested context;
- \(\Delta\) be measured transient overhead not represented in those terms.

If the residency manager maintains the invariant

\[
\sum_i \mathbf{1}[m_i\text{ resident}] \le 1,
\]

then

\[
M_{peak} \le B+G+R+\max_i F_i+\Delta.
\]

**Proof.** At every time \(t\), either no specialist is resident or exactly one specialist \(i\) is resident. Thus specialist memory at \(t\) is bounded by \(\max_i F_i\). Adding the fixed runtime terms and transient term gives the result. ∎

**Important qualification.** The theorem applies to the defined runtime allocations. A backend that retains old mappings, duplicates model pages, or performs overlapping loads violates the premise unless that overlap is included in \(\Delta\). The measurement harness tests the premise through repeated `/api/ps` samples and fails the run if more than one model appears.

### Proposition 1: Authority non-escalation

Assume (i) authority labels are assigned only by trusted code, (ii) untrusted content cannot modify the effect policy, and (iii) every sensitive sink is completely mediated. For any segment with authority \(a\) and requested effect \(e\notin E(a)\), no model-generated text derived from that segment can cause \(e\) through the mediated runtime.

**Proof.** The sink executes only after the external guard evaluates membership in \(E(a)\). Since \(e\notin E(a)\), the guard rejects the request. The content of the model output does not alter \(a\) or \(E\) under the assumptions. ∎

This is a runtime property, not a guarantee that the model will never produce malicious text.

### Proposition 2: Audit completeness for mediated decisions

If every effect decision calls the receipt function before returning, then each mediated allow or deny outcome has a corresponding structured audit record. This does not guarantee log durability or tamper resistance unless storage and hash-chain requirements are also enforced.

## 8. Reference Artifact

The artifact contains:

- `benchmarks/measure_bounded_agent.py` — live and fake backends, router, unload barrier, memory sampler, result validator, JSON/CSV/SVG writers;
- `benchmarks/bounded_agent_config.example.json` — route profiles and labeled work items;

- CI configuration that runs the harness self-test and unit tests.

### 8.1 Validity rules

A live item is valid only when:

- generation completes without error;
- every sampled resident-model set has cardinality at most one;
- the selected model is observed resident at least once;
- the requested route exists;
- when strict routing is enabled, the selected route matches the labeled expected route.

The harness exits nonzero on invalid results. It stores output hashes rather than full generated text by default, reducing accidental retention of sensitive prompts while still making reruns comparable.

### 8.2 Artifacts

Each live run writes:

- `results.json`: configuration, platform, summary, per-item metrics, and validity flags;
- `memory_samples.csv`: time, host memory, process RSS, and loaded-model sets;
- `ram_over_time.svg`: dependency-free figure generated from raw samples.

## 9. Evaluation Questions

A complete empirical evaluation should answer:

- **RQ1 — Residency:** Does the implementation maintain at most one resident specialist, and what is measured peak RAM?
- **RQ2 — Performance:** What are median and tail latency, token throughput, and model-switch penalty by route?
- **RQ3 — Guardrails:** How much do deterministic gates reduce unauthorized tool, memory, and policy effects, and what benign-task false-positive cost do they introduce?
- **RQ4 — Capability:** How do routed specialists compare with the best single resident model on public code, reasoning, and general-language tasks under the same memory budget?
- **RQ5 — Routing:** How does a deterministic router compare with learned and oracle routing?
- **RQ6 — Ablation:** What changes when route-specific decoding, unload barriers, authority labels, confirmation gates, or detectors are removed?

### 9.1 Required protocol

Hardware claims should include CPU, GPU/NPU, RAM, OS, backend version, quantization, context size, warm/cold state, sampling interval, and complete model names. Capability benchmarks must use fresh context per item and fixed seeds where supported. Security results must report attack success, utility under no attack, false positives, and adaptive attacks—not only known signatures.

## 11. Threats to Validity

### 11.1 Memory measurement

Host-used memory is noisy because unrelated processes allocate memory. Process RSS may omit memory held by a separate model-server process. Backend-reported loaded models are therefore the primary residency check, while system memory is a coarse end-to-end measurement. A stronger study should also sample the model-server process tree and GPU/NPU memory.

### 11.2 Routing evaluation

The included heuristic router is intentionally simple. Keyword overlap can misroute ambiguous prompts, and a learned router may outperform it. Oracle routing must be defined using held-out task labels rather than post-hoc selection on final answers.

### 11.3 Security evaluation

An authority gate prevents effects only at mediated sinks. It does not prevent misleading text, social engineering of the user, denial of service, side channels, vulnerabilities in the tool implementation, or compromised trusted code. Pattern detectors remain bypassable and are treated as secondary signals.

### 11.4 Capability evaluation

Specialization may help some tasks and harm others. A routed collection must be compared against a single model under equal memory, quantization, context, and compute budgets. Anecdotes are insufficient.

### 11.5 Local-first privacy

Local execution reduces network disclosure but does not guarantee privacy. Logs, caches, embeddings, crash dumps, model-server history, and local malware can expose data. Local-first is an architectural preference, not a complete privacy proof.

## 13. Discussion

The architecture does not attempt to solve limited model capability by hiding it. Instead, it decomposes the problem. Specialist routing allocates the available model budget to the task most likely to benefit. Serialized residency turns a collection of models into a memory-bounded service at the cost of switch latency. Route-aware parameters reduce avoidable cross-task decoding failures. Deterministic guards reduce the consequences of model error by refusing to treat model text as authorization.

This decomposition is valuable because model quality and system authority are different variables. A stronger model can still be prompt-injected. A weaker model can still be prevented from deleting a file. Conversely, guardrails do not make a weak answer correct. Reliable local agents require both capability evaluation and consequence containment.

The architecture also clarifies where future optimization belongs. If switching dominates latency, use bounded keep-alive windows, task batching, adapter swapping, or a shared base model with route-specific adapters. If routing quality is poor, train a small router. If a specialist exceeds memory, use quantization, offloading, or an inference engine such as PowerInfer. If security utility is low, refine capabilities and confirmation policies rather than delegating authority decisions back to the model.

## 14. Conclusion

A local multi-capability agent does not need every specialist resident at once, and a language model does not need final authority over side effects. Serialized specialist routing provides a simple conditional memory bound: under complete unload-before-load behavior, peak specialist memory is determined by the largest active model rather than the sum of all models. Deterministic authority guardrails provide a separate conditional security property: under trusted labeling and complete mediation, untrusted context cannot authorize disallowed effects merely by persuading the model.

The accompanying artifact turns these claims into testable conditions. It records residency instead of assuming it, fails invalid runs, and separates synthetic harness validation from real model measurements. The current evidence supports the architecture and regression behavior of the guardrail component, but not a hardware-performance or universal-capability claim. That distinction is the central methodological point of this report: bounded agents should be evaluated through explicit invariants, raw measurements, and externally enforced contracts—not persuasive demonstrations alone.
